# 캐릭터 상품 전환 시 수정사항 체크리스트

## 목적

현재 MVP는 `단일 상품 = 단일 가격 = 단일 수량 선택` 구조입니다.
캐릭터 상품으로 운영하려면 옵션, SKU, 재고, 노출 방식이 더 세분화되어야 하므로 아래 항목을 점검해야 합니다.

---

## 현재 구조 진단

현재 상품 구조는 아래 전제를 가지고 있습니다.

- 상품당 이름 1개
- 상품당 가격 1개
- 상품당 이미지 1개
- 옵션 없음
- 캐릭터 구분 없음
- 사이즈/색상/버전 없음
- 재고 없음
- 주문 시 상품 스냅샷도 단일 상품명/단일 가격만 저장

즉, 아래 같은 캐릭터 상품 시나리오를 바로 처리하기 어렵습니다.

- 같은 캐릭터의 키링 / 인형 / 머그컵 / 스티커 세트
- 같은 상품의 캐릭터 버전별 가격 차이
- 랜덤 굿즈 / 한정판 / 시즌 상품
- 캐릭터별 재고 수량 관리
- 옵션 선택 후 주문

---

## 필수 수정 체크리스트

### 1. 상품 분류 체계 추가

- [ ] `캐릭터명` 필드 추가
- [ ] `상품 카테고리` 필드 추가
- [ ] `시리즈/라인업` 필드 추가 여부 결정
- [ ] `한정판 여부` 필드 추가 여부 결정
- [ ] `추천 상품` 또는 `신상품` 뱃지 필드 추가 여부 결정

권장 예시:

- 캐릭터명: `춘식이`
- 카테고리: `인형`, `키링`, `문구`, `컵`, `의류`
- 시리즈: `봄 에디션`, `크리스마스`, `랜덤 피규어`

### 2. 상품 옵션 / SKU 구조로 변경

- [ ] 상품과 옵션을 분리할지 결정
- [ ] `product_variants` 테이블 추가 검토
- [ ] 옵션별 이름 추가
- [ ] 옵션별 가격 추가
- [ ] 옵션별 재고 추가
- [ ] 옵션별 이미지 추가 여부 결정
- [ ] 옵션별 노출 여부 추가

현재 구조는 `products.price` 하나만 사용하므로,
아래 같은 구조는 지원되지 않습니다.

- 춘식이 키링 - 노랑: 9,900원
- 춘식이 키링 - 핑크: 10,900원
- 춘식이 인형 - 소형: 19,900원
- 춘식이 인형 - 대형: 29,900원

권장 방향:

- `products`: 전시용 대표 상품
- `product_variants`: 실제 주문되는 SKU

### 3. 재고 관리 추가

- [ ] 상품 단위 재고가 필요한지 결정
- [ ] 옵션 단위 재고가 필요한지 결정
- [ ] 품절 상태 필드 추가
- [ ] 안전재고 기준 추가 여부 결정
- [ ] 주문 시 재고 차감 로직 추가
- [ ] 관리자 재고 수정 UI 추가

캐릭터 상품은 품절 빈도가 높아서 재고 없이 운영하면 주문 누락 가능성이 큽니다.

### 4. 관리자 상품 등록 폼 확장

- [ ] 캐릭터명 입력 필드 추가
- [ ] 카테고리 선택 필드 추가
- [ ] 다중 이미지 등록 여부 결정
- [ ] 옵션 행 추가 UI 필요 여부 결정
- [ ] 대표 이미지 / 상세 이미지 분리 여부 결정
- [ ] 품절 상태 직접 변경 기능 추가 여부 결정

현재 관리자 상품 폼은 아래 항목만 지원합니다.

- 상품명
- 설명
- 판매가
- 도매 원가
- 도매처
- 정렬 순서
- 노출 여부

캐릭터 상품 운영에는 정보량이 부족합니다.

### 5. 키오스크 목록 화면 수정

- [ ] 캐릭터별 묶음 노출 여부 결정
- [ ] 카테고리 필터 필요 여부 결정
- [ ] 캐릭터 필터 필요 여부 결정
- [ ] 추천 / 신상품 / 한정판 뱃지 표시 여부 결정
- [ ] 품절 상품 노출 정책 결정

예시 정책:

- 품절 상품은 숨김
- 또는 품절 배지와 함께 노출 유지

### 6. 상품 상세 화면 수정

- [ ] 옵션 선택 UI 추가
- [ ] 캐릭터 버전 선택 UI 추가
- [ ] 선택 옵션에 따라 가격 변경
- [ ] 선택 옵션에 따라 이미지 변경 여부 결정
- [ ] 품절 옵션 비활성화 처리
- [ ] 상세 이미지 여러 장 노출 여부 결정

현재는 수량만 선택 가능하므로 캐릭터 상품 판매에는 부족합니다.

### 7. 주문 데이터 구조 변경

- [ ] 주문에 `variant_id` 저장
- [ ] 주문에 `variant_name` 저장
- [ ] 주문에 `character_name` 스냅샷 저장
- [ ] 주문에 `category_name` 스냅샷 저장 여부 결정
- [ ] 옵션별 가격 스냅샷 저장

현재 주문에는 아래 정도만 저장됩니다.

- `product_id`
- `product_name`
- `product_price`
- `quantity`

옵션형 캐릭터 상품은 이 정보만으로 주문 복원이 부족합니다.

### 8. 도매처 연동 구조 재검토

- [ ] 캐릭터별 도매처가 다른지 확인
- [ ] 옵션별 도매처 분기가 필요한지 확인
- [ ] 동일 상품 내 옵션별 공급처 분리 필요 여부 확인

현재는 상품당 도매처 1개 연결 구조입니다.
캐릭터 굿즈는 공급처가 더 세분화될 수 있습니다.

### 9. 이미지 정책 강화

- [ ] 대표 썸네일 이미지
- [ ] 상세 이미지 여러 장
- [ ] 투명 배경 PNG 필요 여부
- [ ] 정사각형 썸네일 규격 고정 여부
- [ ] 옵션별 이미지 분리 여부

캐릭터 상품은 이미지 완성도가 구매 전환에 직접 영향이 큽니다.

### 10. 노출/숨김 정책 정교화

- [ ] 전체 숨김
- [ ] 옵션 일부만 숨김
- [ ] 품절 시 자동 숨김 여부
- [ ] 예약 판매 상태 추가 여부
- [ ] 판매 종료 상태 추가 여부

현재는 `is_active` 하나로만 제어합니다.
캐릭터 상품 운영에는 상태 구분이 더 필요할 수 있습니다.

## DB 수정 권장안

### A안: 최소 확장

현재 `products` 테이블에 필드만 추가

- `character_name`
- `category_name`
- `stock_quantity`
- `is_sold_out`
- `badge_type`

장점:

- 구현이 빠름
- MVP 확장에 유리

단점:

- 옵션 상품 대응이 어려움
- SKU 관리가 불완전함

### B안: 권장 구조

`products` + `product_variants` 2단 구조

예시:

- `products`: 대표 상품 정보
- `product_variants`: 옵션별 실제 판매 단위

`products` 예시 필드:

- id
- name
- description
- character_name
- category_name
- image_url
- wholesaler_id
- is_active

`product_variants` 예시 필드:

- id
- product_id
- variant_name
- sku
- price
- wholesale_price
- stock_quantity
- is_active
- image_url

장점:

- 캐릭터 상품 운영에 맞음
- 옵션/재고/가격 분리 가능
- 주문 정확도 높음

단점:

- 화면/API 수정 범위가 큼

## 결론

현재 구조는 일반 단일상품 판매 MVP로는 적절하지만,
캐릭터 상품 운영에는 아래 3가지가 가장 먼저 부족합니다.

1. 옵션/SKU 구조 부재
2. 재고/품절 관리 부재
3. 캐릭터/카테고리 메타정보 부재

캐릭터 상품이 단순 굿즈 1종씩만 등록되는 수준이면 `A안`으로도 시작할 수 있습니다.
하지만 캐릭터별 옵션, 사이즈, 색상, 한정판, 재고 운영까지 고려하면 `B안(products + product_variants)`로 가는 것이 맞습니다.
